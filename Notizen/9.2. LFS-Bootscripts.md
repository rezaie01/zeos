- Installationshinweis: Einzeiliger Installationsbefehl.

- Kurzbeschreibungen der einzelnen Bootscripte (Funktion pro Skript).

### Minimalistische Zusammenfassung

Das Paket liefert eine Sammlung von Skripten, die den Übergang vom Kernel ins Nutzersystem regeln: Mounten von Dateisystemen, Gerätevorbereitung, Swap-Aktivierung, Uhrsetzen, Netzwerkinitialisierung, Start/Stop von Daemons und sauberes Herunterfahren/Reboot. Installation erfolgt mit dem Befehl:

make install

Installierte Skripte (exakt übernommen):  
checkfs, cleanfs, console, functions, halt, ifdown, ifup, localnet, modules, mountfs, mountvirtfs, network, rc, reboot, sendsignals, setclock, ipv4-static, swap, sysctl, sysklogd, template, udev, and udev_retry

Installierte Verzeichnisse (exakt übernommen):  
/etc/rc.d, /etc/init.d (symbolic link), /etc/sysconfig, /lib/services, /lib/lsb (symbolic link)

Kurzbeschreibungen der Bootscripte (exakt übernommen, Kurz-Erklärung nur wenn funktional relevant):

- checkfs — Checks the integrity of the file systems before they are mounted (with the exception of journaling and network-based file systems)
    
- cleanfs — Removes files that should not be preserved between reboots, such as those in /run/ and /var/lock/; it re-creates /run/utmp and removes the possibly present /etc/nologin, /fastboot, and /forcefsck files
    
- console — Loads the correct keymap table for the desired keyboard layout; it also sets the screen font
    
- functions — Contains common functions, such as error and status checking, that are used by several bootscripts
    
- halt — Halts the system
    
- ifdown — Stops a network device
    
- ifup — Initializes a network device
    
- localnet — Sets up the system's hostname and local loopback device
    
- modules — Loads kernel modules listed in /etc/sysconfig/modules, using arguments that are also given there
    
- mountfs — Mounts all file systems, except those that are marked noauto, or are network based
    
- mountvirtfs — Mounts virtual kernel file systems, such as proc
    
- network — Sets up network interfaces, such as network cards, and sets up the default gateway (where applicable)
    
- rc — The master run-level control script; it is responsible for running all the other bootscripts one-by-one, in a sequence determined by the names of the symbolic links to those other bootscripts
    
- reboot — Reboots the system
    
- sendsignals — Makes sure every process is terminated before the system reboots or halts
    
- setclock — Resets the system clock to local time if the hardware clock is not set to UTC
    
- ipv4-static — Provides the functionality needed to assign a static Internet Protocol (IP) address to a network interface
    
- swap — Enables and disables swap files and partitions
    
- sysctl — Loads system configuration values from /etc/sysctl. conf, if that file exists, into the running kernel
    
- sysklogd — Starts and stops the system and kernel log daemons
    
- template — A template to create custom bootscripts for other daemons
    
- udev — Prepares the /dev directory and starts the udev daemon
    
- udev_retry — Retries failed udev uevents, and copies generated rules files from /run/udev to /etc/udev/rules. d if required
    

Hinweis: Einige Skripte referenzieren Konfigurationspfade (z. B. /etc/sysconfig/modules, /etc/[sysctl.conf](https://sysctl.conf), /run/, /etc/udev/rules.d). Diese Dateien/Verzeichnisse dienen als Eingabe für die Bootscripte und bestimmen Laufzeitverhalten.

### Verständnisnotizen — Was ich mir unbedingt merken sollte

1. LFS-Bootscripts bündeln alle praktischen Start-/Stopp-Schritte in kleinen, spezialisierten Skripten, die einzeln test- und tauschbar sind.
    
2. rc ist der Master-Controller: Reihenfolge wird über die Namen/symlinks der Bootscripte gesteuert.
    
3. setclock und checkfs sind integrierte Sicherheits- und Konsistenzschritte (Uhr setzen; Dateisystemintegrität prüfen).
    
4. localnet, ifup/ifdown und ipv4-static regeln die minimale Netzwerkinitialisierung; hostname und Loopback werden vom localnet-Skript gesetzt.
    
5. udev/udev_retry kümmern sich um /dev-Erzeugung und fehlertolerante Behandlung von Geräteevents.
    
6. Viele Skripte lesen Konfigurationsdateien an festen Pfaden; Anpassung erfolgt durch Ändern dieser Dateien (z. B. /etc/sysconfig/modules, /etc/[sysctl.conf](https://sysctl.conf)).

### Fragen

1. Sind die Bootskripte standard über alle Linuxe, oder gehören sie nur zu LFS?
    

- Kurz: Nein — die traditionellen Bootskripte (Shell-Skripte in /etc/init.d, /etc/rc.d, etc.), wie sie in LFS / vanilla Unix-Style verwendet werden, sind nicht bei allen Linux-Distributionen gleich standardisiert. LFS liefert diese klassische, script‑basierte Struktur als Teil des Lern- und Aufbauprozesses, weil sie die Funktionsweise der Bootsequenz klar macht.
    

2. Wenn nur LFS, was haben die anderen Distros stattdessen?
    

- Moderne verbreitete Init-Systeme und deren Kerneigenschaften:
    
    - systemd
        
        - Struktur: unit‑Dateien statt Shell-Skripte; zentrale Daemon‑Architektur.
            
        - Features: Abhängigkeitsauflösung, paralleler Start, Socket‑ und D‑Bus‑Aktivierung, cgroup‑Management.
            
        - Verwaltung: systemctl; Units liegen meist unter /lib/systemd/system und /etc/systemd/system.
            
    - SysV init (klassische init‑Skripte)
        
        - Struktur: /etc/init.d-Skripte und Runlevel‑Symlinks in /etc/rc?.d.
            
        - Merkmale: sequenzieller Start, einfache Skripte, wenig Infrastruktur für Abhängigkeiten.
            
        - Verwendet noch in einigen minimalistischen oder älteren Distros.
            
    - OpenRC
        
        - Struktur: Shell‑Skripte mit deklarativen Metadaten; init‑daemon übernimmt Start/Stop.
            
        - Merkmale: Init‑Skripte + Abhängigkeitsgraph; leichter als systemd; genutzt von Gentoo, Alpine (teilweise).
            
    - runit
        
        - Struktur: einfache, zustandslose Dienstverzeichnisse; drei Phasen (init, run, finish).
            
        - Merkmale: sehr klein, schnell, prozessorientiert; genutzt von Void Linux.
            
    - s6 / s6‑init
        
        - Struktur: Prozessüberwachung + Dienstverzeichnisse; designfokussiert auf Zuverlässigkeit und Minimalismus.
            
        - Merkmale: modular, robust, in einigen sicherheitsbewussten/minimalen Distros eingesetzt.
            
    - busybox init / initramfs‑basierte Lösungen
        
        - Struktur: minimaler Init in eingebetteten Systemen; oft nur für initramfs-Phase oder kleine Geräte.
            
- Wichtige Unterschiede zur LFS/Script‑Methode:
    
    - Sprache/Form: systemd nutzt deklarative Unit‑Dateien; LFS‑Skripte sind Shell‑Skripte.
        
    - Abhängigkeiten: systemd/OpenRC handhaben Abhängigkeiten; traditionelle SysV-Skripte oft manuell/Runlevel‑based.
        
    - Parallelität: systemd startet Services parallel, SysV meistens seriell.
        
    - Management‑Tools: systemctl (systemd), rc-service/OpenRC, sv (runit), init scripts mit service/insserv-Aufrufen (SysV).
        
    - Sichtbarkeit/Standardisierung: systemd hat sich in vielen großen Distros durchgesetzt, andere behalten alternative Initsysteme aus Design‑ oder Philosophiegründen.
        
- Praktische Folgerung für LFS‑Notizen:
    
    - Notiere, dass LFS bewusst die script‑basierte Init‑Struktur lehrt; vermerke für jede Distribution das übliche Init‑System (z. B. Debian/Ubuntu — systemd; CentOS/RHEL — systemd; Gentoo — OpenRC standardmäßig; Void — runit; Alpine — OpenRC) und die relevanten Orte/Befehle zum Verwalten von Services (z. B. systemctl, rc-service, sv, /etc/init.d).

### Glossar

- rc  
    Master-Skript, das alle anderen Bootscripte sequenziell ausführt.
    
- ifup / ifdown  
    Skripte zum Hoch- und Herunterfahren von Netzwerkinterfaces.
    
- udev / udev_retry  
    Erzeugung/Verwaltung von Gerätedateien in /dev; Wiederholungsmechanismus für fehlgeschlagene Events.
    
- setclock  
    Skript zum Setzen der Systemuhr (beachtet Hardware-Uhr vs. UTC-Flag).
    
- checkfs / cleanfs  
    Integritätsprüfung von Dateisystemen; bereinigt temporäre Dateien zwischen Reboots.
    
- /etc/sysconfig/modules  
    Datei/Ort, aus dem modules Skript die zu ladenden Kernelmodule liest.
    
- /etc/[sysctl.conf](https://sysctl.conf)  
    Konfigurationsdatei mit Kernelparametern, geladen vom sysctl-Skript.

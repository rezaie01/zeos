- Überblick über Aufgaben beim Systemstart: Dateisysteme mounten, Geräte initialisieren, Swap aktivieren, Systemuhr setzen, Netzwerk hochfahren, Daemons starten.
- Vorstellung des klassischen System V Bootablaufs mit init und rc-Skripten.

- Vorteile und Nachteile des System V Ansatzes: etabliert und anpassbar; serielles Abarbeiten und fehlende moderne Features.

---

### Minimalistische Zusammenfassung

Bootvorgang listet Reihenfolge von Pflichtaufgaben auf: virtuelle und reale Dateisysteme mounten, Geräte initialisieren, Dateisystemintegrität prüfen, Swap mounten/aktivieren, Systemuhr setzen, Netzwerk hochfahren, benötigte Hintergrundprozesse (Daemons) starten, und benutzerdefinierte Tasks ausführen. Diese Schritte müssen in sinnvollem, möglichst schnellem Ablauf organisiert werden.

System V ist die klassische Methode seit ca. 1983. Kernprinzip: ein kleines Programm, init, startet grundlegende Prozesse (z. B. Login via getty) und führt ein Master-Skript aus (typischerweise rc). Dieses Master-Skript steuert die Ausführung einer Reihe weiterer Initialisierungsskripte. Die Steuerdatei für init ist

/etc/inittab

Runlevels (genau übernommen):  
0 - halt  
1 - Single user mode  
2 - User definable  
3 - Full multiuser mode  
4 - User definable  
5 - Full multiuser mode with display manager  
6 -reboot

Üblicher Default ist Runlevel 3 oder 5.

Vorteile: etabliertes, gut verstandenes System; leicht anpassbar.  
Nachteile: potentiell langsameres Booten (Beispiel: 8–12 Sekunden Kernel->Login auf einem mittleren Basis-LFS-System; Netzwerk meist ~2 Sekunden nach Login verfügbar); serielle Abarbeitung von Tasks (ein verzögertes Element bremst das ganze Booten); keine direkte Unterstützung moderner Mechanismen wie cgroups; Hinzufügen von Skripten erfordert manuelle, statische Sequenzierung.

---

### Verständnisnotizen — Was ich mir merken sollte

1. Boot ist mehrschichtig: Kernelstart ist nur der Anfang; Nutzerland-Aufgaben müssen geordnet abgearbeitet werden.
2. init + rc-Skripte bilden beim System V das zentrale Steuerungsmodell; /etc/inittab definiert Verhalten und Runlevels.
3. Runlevels sind einfache numerische Zielzustände; typische Default-Levels sind 3 (text) oder 5 (mit Display Manager).
4. Serialität im Boot kann zu Verzögerungen führen; moderne Features wie cgroups fehlen nativ.
5. Anpassung ist einfach möglich, aber Reihenfolge der Skripte muss manuell bedacht werden.

### Glossar

- init  
    Kleines Benutzerland-Programm, das den Bootprozess steuert und Startskripte ausführt.
    
- rc  
    Master-Skript, das weitere Initialisierungsskripte startet.
    
- /etc/inittab  
    Steuerdatei für init; definiert Runlevels und Startverhalten.
    
- Runlevel  
    Numerische Definition eines Zielzustands des Systems (z. B. Single-User, Multi-User, Reboot).
    
- getty  
    Programm, das Login-Terminals bereitstellt; von init gestartet.
    
- Daemon  
    Hintergrundprozess, der typischerweise beim Systemstart aktiviert wird.
    
- Swap  
    Auslagerungsbereich (Partition oder Datei), der beim Booten aktiviert werden kann.
